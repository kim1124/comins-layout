# Contract v1.2 Required Security And 0.1.3 Release Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Require the existing `Sensitive data` GitHub check on `main`, prepare and fully verify `comins-grid-layout@0.1.3`, then publish it as npm `latest` through trusted staged publishing.

**Architecture:** Preserve the current package and workflow design. Mutate only the existing GitHub ruleset required-check list, prepare patch-release metadata in the repository, validate the exact package archive locally and remotely, and use the existing OIDC `npm stage publish` workflow. The maintainer completes the proof-of-presence approval in npm; Codex then verifies the public artifact and records final evidence.

**Tech Stack:** React 19, TypeScript 6, GridStack 12, Vite 8, Vitest 4, Playwright 1.61, GitHub Actions, GitHub Rulesets API, npm 11 staged publishing, Gitleaks 8.30.1.

## Global Constraints

- Keep the root managed guidance block byte-identical to the governance Contract v1.2 template.
- Preserve public APIs, product source, CSS, dependencies, workflow structure, existing ruleset conditions, and all non-status-check rules.
- Never print detected sensitive values, personal identities, value-derived fingerprints, or scanner details; reduce scanner failures to constant messages.
- Do not publish directly from the workstation, push directly to `main`, create a Git tag, or create a GitHub Release.
- Treat npm staging as non-public until the maintainer approves it with 2FA and registry verification succeeds.
- Stop before merge, stage approval, or release completion if the exact expected SHA, version, required check, provenance, or artifact integrity cannot be established.
