# Contract v1.2 Required Security And 0.1.3 Release Design

## Goal

Close the remaining Comins Contract v1.2 enforcement gap, verify the complete implemented Grid Layout package, and publish `comins-grid-layout@0.1.3` as `latest` through the existing trusted staged-publishing workflow.

## Confirmed Decision

- Use the recommended provider-enforced approach: add the existing `Sensitive data` GitHub Actions check to the active `main-protection` ruleset alongside `verify`.
- Publish the next stable Semantic Versioning patch, `0.1.3`, with npm dist-tag `latest`.
- Use the existing GitHub `npm` environment, OIDC trusted publisher, and `npm stage publish` approval flow.
- Do not create a Git tag or GitHub Release.

## Verified Starting State

- Governance `COMINS_CONTRACT.md` is v1.2 and `SENSITIVE_DATA_STANDARD.md` is the active common security source.
- The root `AGENTS.md` managed block matches `templates/module/AGENTS.md` exactly and preserves module-specific guidance outside the block.
- PR #7 has been merged into `main`; local `main`, `origin/main`, and commit `b0b019d` agree.
- The local hooks, Gitleaks `8.30.1`, public-identity gate, security CI, exact package-artifact gate, consumer test, and trusted stage workflow are already implemented.
- The active `main-protection` ruleset requires only `verify`; the separate `Sensitive data` check passes but is not yet required.
- Public npm `latest` is `0.1.2`, built from commit `4dd2512`. The selector-safe widget ID fix and Contract v1.2 enforcement were merged after that release.

## Scope

### Repository Changes

- Change `package.json` and the root package entry in `package-lock.json` from `0.1.2` to `0.1.3`.
- Add a `0.1.3` entry to `CHANGELOG.md` describing the selector-safe widget restoration fix and the Contract v1.2 release gates without claiming a public API change.
- Add release preparation and verification evidence to `reports/2026-07-22.md` before merge, then append public-release evidence after npm publication.
- Keep product source, public types, CSS, dependency versions, and workflow structure unchanged unless verification exposes a release blocker.

### GitHub Enforcement Change

- Read the active ruleset immediately before mutation and require the expected current state: active `main-protection`, branch target, and existing required check `verify`.
- Add `Sensitive data` to the same required-status-check rule while preserving every other ruleset condition and rule.
- Read the ruleset again after mutation and require both `Sensitive data` and `verify`.
- Do not weaken deletion protection, force-push protection, pull-request requirements, strict update requirements, bypass actors, or any unrelated provider setting.

### Release Operations

- Push one release branch and open one pull request targeting `main`.
- Require both `Sensitive data` and `verify` to pass on the release pull request before merge.
- Merge only the reviewed release pull request; do not push directly to `main`.
- Dispatch `Stage npm package` from the exact merged `main` commit with version input `0.1.3`.
- Treat successful workflow completion as a staged package, not as proof of public publication.
- Complete the npm approval step through the maintainer-authenticated npm UI, then verify registry metadata, `latest`, provenance, and the downloaded public tarball.

## Verification Design

### Local Package Gate

Run, in order:

1. `npm run test:security`
2. `npm run verify:full`
3. `npm run test:consumer`
4. `npm run verify:package-artifact`
5. Extract the single generated archive and scan it with pinned Gitleaks `8.30.1` using `.gitleaks.toml`.
6. Parse both workflow YAML files, run `git diff --check`, and confirm no generated archive or raw scanner output remains tracked or untracked.

The Playwright gate must cover all existing Chromium, mobile Chromium, and isolated 100-widget resource-stability cases. Intentional desktop-input skips in the mobile project remain accepted only when their count and reasons match the current test contract.

### Remote Gate

- Confirm the release pull request is mergeable and current with `main`.
- Confirm `Sensitive data` and `verify` both report success for the exact head SHA.
- After merge, confirm the merged `main` SHA is the commit used by the stage workflow.
- Confirm the workflow completes version validation, immutable install, full browser verification, consumer verification, exact artifact inspection, extracted-package Gitleaks scanning, artifact transfer, and `npm stage publish`.

### Public Package Gate

- Confirm npm exposes version `0.1.3` and `latest=0.1.3`.
- Confirm the public `gitHead` equals the exact merged `main` commit used by the workflow.
- Confirm SLSA provenance is present.
- Download the public tarball to an isolated temporary directory, compare its integrity with registry metadata, inspect its file list, and scan the extracted contents without printing sensitive values.
- Perform an isolated consumer install from the public `0.1.3` package and import both the package and stylesheet exports.

## Failure Handling

- Any local security, build, browser, consumer, or package-artifact failure blocks the release branch push.
- Any failed or missing required GitHub check blocks merge.
- Any ruleset mismatch blocks provider mutation; do not replace the ruleset from stale local assumptions.
- Any staged-package mismatch blocks npm approval.
- Any registry, provenance, integrity, or public consumer failure is reported as a release incident. Do not reuse `0.1.3`; prepare a later patch only after diagnosis and approval.
- Scanner output remains captured and reduced to constant failure messages. Sensitive matches, identities, fingerprints, and paths are never copied into reports or conversation output.

## Excluded Work

- Git tags and GitHub Releases.
- Removal of historical npm `0.1.0` or `0.1.1` and npm account metadata changes.
- Legacy Git/provider cleanup already tracked through separate audits.
- Keyboard move/resize, touch drag/resize, new product features, dependency upgrades, and unrelated refactoring.

## Acceptance Criteria

- Root guidance remains an exact Contract v1.2 managed-block match.
- `main-protection` requires both `Sensitive data` and `verify` without weakening any other rule.
- `0.1.3` contains the same public API and package files contract as `0.1.2`, plus the already-merged selector-safe runtime fix.
- Local full, consumer, package, PR, merge, staged publish, npm approval, registry, provenance, tarball, and public consumer checks all pass with recorded evidence.
- `latest` resolves to `0.1.3`, no tag or GitHub Release is created, and the repository is clean at handoff.
